# AGENTS.md

Этот файл задает правила работы агента в проекте.
Цель: максимальная скорость, предсказуемость, качество кода и контроль изменений.

## 1. Режим работы агента

- Агент выполняет задачи end-to-end: анализ, правки, проверка, commit, push.
- Агент не останавливается на плане, если можно сделать изменения сразу.
- Агент не создает фоновые автозадачи (scheduler, cron, services) без прямой команды пользователя.
- Агент не делает "автопуш каждые N минут".
- Пуш делается только по факту реальных изменений в рамках задачи.
- Если задача рискованная или неоднозначная, агент коротко фиксирует варианты и просит выбор.

## 2. Git и публикация изменений

- Основная ветка: `master` (или актуальная ветка, которую указал пользователь).
- Если `git commit` не проходит из-за отсутствия `user.name`/`user.email`, агент настраивает их автоматически и продолжает работу без запроса к пользователю.
- Порядок автоконфигурации git identity:
  - сначала локально в репозитории (`git config user.name`, `git config user.email`);
  - значения брать из уже известных данных пользователя или из глобального git-конфига;
  - при отсутствии значений использовать дефолт пользователя: `user.name=Nikita`, `user.email=stithc92@gmail.com`;
  - после успешного commit при необходимости синхронизировать эти значения в глобальный git-конфиг.
- Для каждой завершенной логической задачи:
  - `git add -A`
  - осмысленный `git commit`
  - `git push`
- Формат commit message:
  - `feat: ...` для нового поведения
  - `fix: ...` для исправлений
  - `refactor: ...` для переработки без изменения поведения
  - `test: ...` для тестов
  - `docs: ...` для документации
  - `chore: ...` для сервисных изменений
- Агент не делает `git reset --hard`, `git checkout --`, force-push и другие разрушительные действия без явного разрешения.
- Агент не трогает несвязанные изменения пользователя.

## 3. Качество кода (обязательно)

- Любая правка должна быть:
  - минимальной по поверхности изменений;
  - понятной при чтении;
  - совместимой с текущей архитектурой проекта;
  - без "магии" и скрытых сайд-эффектов.
- Если есть выбор между быстрым хаком и устойчивым решением, выбирать устойчивое.
- Дублирование логики сокращать. Повторяющийся код выносить в функции/модули.
- Публичные функции и сложные участки должны иметь короткие, практичные комментарии.
- Не добавлять зависимости без явной пользы.

## 4. Тестирование и проверки перед пушем

- Перед commit агент обязан прогнать релевантные проверки:
  - минимум: целевые тесты по измененным модулям;
  - лучше: полный прогон `pytest`, если не слишком долго.
- Если автотестов не хватает:
  - добавить тесты для новой логики/исправления;
  - или явно зафиксировать, что проверить вручную.
- Если тесты не запускаются из-за окружения, агент пишет это прямо в отчете и не скрывает проблему.

## 5. Python-стандарты проекта

- Версия Python: 3.10+.
- Стиль:
  - понятные имена функций и переменных;
  - небольшие функции с одной ответственностью;
  - ранние возвраты вместо глубокой вложенности;
  - явная обработка ошибок с полезными сообщениями.
- Импорты должны быть чистыми и неиспользуемые импорты удаляются.
- Платформенную специфику Windows (`winsound`, `ctypes`, `schtasks`) изолировать, чтобы она не ломала остальной код.

## 6. Наблюдаемость и контроль

- Ошибки должны быть диагностируемыми:
  - понятный текст ошибки;
  - контекст, что именно делалось;
  - безопасный fallback, где это возможно.
- При изменении пользовательских сценариев обновлять:
  - `README.md` (если изменился запуск, поведение или настройка);
  - примеры команд/инструкций.
- Для заметных решений фиксировать краткую запись в описании commit.

## 7. Безопасность

- Секреты (токены, ключи, пароли) не хардкодить в коде и не коммитить.
- Использовать переменные окружения и/или локальные конфиги вне git.
- При утечке токена сразу рекомендовать ротацию и удаление старого ключа.

## 8. Шаблон рабочего цикла на задачу

1. Уточнить цель и критерий готовности (если неочевидно).
2. Посмотреть релевантные файлы и зависимости.
3. Внести минимально необходимые изменения.
4. Запустить тесты/проверки.
5. Коротко описать, что изменено и почему.
6. Сделать commit и push.

## 9. Definition of Done

Задача считается завершенной, когда:

- код реализует требуемое поведение;
- тесты (или релевантные проверки) пройдены;
- нет очевидных регрессий в соседних сценариях;
- изменения закоммичены и запушены;
- пользователь получил краткий отчет: что сделано, что проверено, что осталось.

## 10. Политика общения с пользователем

- Писать коротко, по делу, без воды.
- Не скрывать риски и ограничения.
- Предлагать следующий шаг только когда это действительно полезно.
- Всегда давать итог, который можно проверить: файл, команда, commit, push.

## 11. Правила по кириллице и кодировкам

- В коде (имена переменных, функций, классов, модулей, файлов) использовать только ASCII/латиницу.
- Кириллица разрешена в:
  - пользовательских сообщениях интерфейса;
  - документации (`README.md`, `AGENTS.md`);
  - тестовых данных, где важны русские кейсы.
- Если строка уходит во внешний API, лог, JSON или файл, учитывать кодировку и явно использовать UTF-8.
- Для всех текстовых исходников и конфигов по умолчанию использовать UTF-8 (включая Python, PHP, Blade, JS/TS, HTML/CSS, SQL, YAML, JSON, Markdown, `.env`-шаблоны).
- Не смешивать без причины русский и английский в одном пользовательском сообщении.
- При добавлении кириллицы в логи/файлы проверять, что текст корректно читается после записи и повторного чтения.
- Перед commit проверять, что не появилось «кракозябр» в измененных файлах.

## 12. Локальная LLM-прослойка (обязательная политика)

- Агент обязан использовать локальную LLM-прослойку автоматически, без действий пользователя.
- Если команда `ollama` недоступна в текущем окружении, агент обязан выполнить self-heal:
  - использовать shim `C:\Users\user\AppData\Local\Microsoft\WindowsApps\ollama.cmd`;
  - при необходимости вызывать бинарник напрямую `C:\Users\user\AppData\Local\Programs\Ollama\ollama.exe`.
- Модели должны храниться на `D:`: `D:\Ollama\models`.
- Cost-First Hybrid (production) по умолчанию:
  - `qwen2.5-coder:7b` — fast pass для кода/шаблонов (q4 profile, лимит 30с);
  - `qwen2.5-coder:7b-instruct-q6_K` — strong local pass, если fast pass не проходит валидацию;
  - `deepseek-r1:8b` — reviewer-проход (короткий список правок, до 8 пунктов);
  - `qwen3:8b` — текстовые/контентные задачи, ТЗ, пояснения;
  - `bge-m3` — embedding-модель для локального RAG;
  - `qwen2.5-coder:14b` — cold standby, не используется в дефолтном роутинге.
- Decision-complete роутинг:
  1. Всегда запускать fast local draft (`qwen2.5-coder:7b`, budget 30s).
  2. При провале acceptance/валидации запускать strong local pass (`qwen2.5-coder:7b-instruct-q6_K`).
  3. Затем запускать локальный reviewer (`deepseek-r1:8b`) для списка до 8 правок без длинных рассуждений.
  4. Cloud fallback включать только если выполнено хотя бы одно:
     - два локальных прохода не дали валидный результат;
     - `ready_sec_local > 120`;
     - `defects_found >= 2` после локальной проверки;
     - задача рискованная: архитектура/безопасность/миграции/6+ файлов.
- Ограничение облака: `max_cloud_calls_per_task = 1` (второй вызов только если тесты красные).
- При cloud fallback отправлять только минимальный контекст: `diff + top-k RAG + ошибка/трейс`, без полного дампа проекта.
- Жесткое правило: для задач генерации кода/верстки/шаблонов сначала локальный черновик через `ollama`, затем доработка.
- Исключения: `ollama` физически недоступна после self-heal или задача явно помечена как `CLOUD_ONLY`.
- Формулировка "локалка не требовалась" запрещена.
- В каждом ответе по задачам генерации обязательна строка аудита:
  - `LLM_LAYER: mode=<LOCAL_FIRST|CLOUD_ONLY>, local_model=<model|none>, first_draft_sec=<n>, ready_sec=<n>, cloud_fallback=<yes|no>, reason=<text>, cloud_calls=<n>, fallback_trigger=<none|validation_failed|time_budget|defects|high_risk>`.
  - правила роутинга и примеры команд находятся в `LOCAL_LLM_ROUTER.md`.

## 13. Автозапуск и безболезненность

- Агент поддерживает автозапуск `ollama serve` через задачу Windows `OllamaServeAtLogon`.
- Watchdog `OllamaWatchdog10m` обязан выполнять health-check каждые 10 минут через `ops/watchdog_ollama.ps1`.
- Если сервер остановлен, агент поднимает его сам перед работой.
- Перед использованием локальной модели агент быстро валидирует готовность:
  - `ollama list`;
  - `ollama run qwen2.5-coder:7b "Reply exactly: READY"`.
- Runtime defaults для 8GB VRAM:
  - `OLLAMA_KEEP_ALIVE=30m`
  - `OLLAMA_MAX_LOADED_MODELS=2`
  - `OLLAMA_NUM_PARALLEL=1`
  - `OLLAMA_CONTEXT_LENGTH=4096`
- Runtime policy:
  - одновременно в памяти: 1 генеративная модель + embedding по требованию;
  - для reviewer-прохода использовать `num_ctx=3072`.
- В отчете агент кратко фиксирует, какая локальная модель применялась и зачем.

## 14. A/B-оценка эффективности (локалка vs без локалки)

- Агент обязан вести измеримую A/B-оценку эффективности режимов `LOCAL_FIRST` и `CLOUD_ONLY`.
- Источник метрик: `LOCAL_LLM_BENCHMARK.md`.
- Для каждого прогона фиксировать минимум:
  - `first_draft_sec`, `ready_sec`, `defects_found`, `cloud_calls`, `cloud_fallback`, `fallback_trigger`, `local_passes`, `retrieval_used`, `retrieval_hit_score`.
- Дополнительно фиксировать стабильность:
  - `watchdog_restarts_day`, `crash_count`, `successful_runs_without_manual`.
- Acceptance gate для нового профиля:
  - `cloud_calls` снижены минимум на 40%;
  - `success_rate` не хуже более чем на 5 п.п.;
  - `defects_found` не выше более чем на +0.3 в среднем;
  - для `layout` и простых `bugfix` — `LOCAL_ACCEPT >= 80%` без облака.
- Гейт на прод-режим:
  - если acceptance выполнен — профиль Cost-First Hybrid становится дефолтным;
  - если нет — вернуть cloud-first для `ui_logic/refactor`, local-first оставить только для `layout`.
